# خطة إعادة تصميم قسم التقنيات المستخدمة في الصفحة الرئيسية

## اسم المهمة
إعادة تصميم قسم التقنيات المستخدمة في الصفحة الرئيسية وتحويله من عرض شعارات تقليدي إلى قسم عصري بعنوان:

**Technology Ecosystem / المنظومة التقنية**

---

## ملاحظة مهمة جدًا

هذه الخطة تخص **Front-end فقط**.

لا يتم تعديل:

- الباك إند.
- قاعدة البيانات.
- API.
- DTOs.
- Migrations.
- Seeders.
- لوحة التحكم.
- طريقة إضافة أو إدارة التقنيات من الإدارة.

المطلوب فقط هو إعادة بناء تجربة العرض داخل الواجهة الأمامية، اعتمادًا على البيانات الحالية القادمة من API.

---

## الهدف من التعديل

القسم الحالي يعرض التقنيات كقائمة شعارات أو Marquee متحرك، وهذا يعطي انطباعًا تقليديًا.

الهدف هو تحويل القسم إلى تجربة بصرية احترافية توضح أن الوكالة لا تستخدم التقنيات كأدوات منفصلة، بل كمنظومة متكاملة لبناء منتجات قابلة للنمو.

بدل أن يقول القسم:

> هذه الأدوات التي نستخدمها

يجب أن يقول:

> هذه المنظومة التي نبني بها منتجات احترافية، مستقرة، وقابلة للتوسع

---

## الملفات المتوقعة للتعديل

الملف الأساسي المتوقع تعديله:

```txt
frontend/src/components/Technologies.tsx
```

وقد يتم تعديل ملفات مساعدة إن وجدت حسب بنية المشروع، مثل:

```txt
frontend/src/components/ui/*
frontend/src/lib/*
frontend/src/styles/*
```

لكن لا يجب إنشاء منطق Backend جديد.

---

## الوضع الحالي المتوقع

قسم التقنيات الحالي غالبًا يعمل بالمنطق التالي:

- جلب التقنيات من API.
- عرض كل التقنيات في Marquee عند اختيار الكل.
- عرض التقنيات كـ Grid عند اختيار تصنيف معين.
- عرض أيقونة التقنية داخل كرت صغير.
- وجود Tooltip بسيط.
- وجود فلاتر حسب التصنيف.

هذا مقبول وظيفيًا، لكنه بصريًا ومنطقيًا لا يعكس وكالة برمجية احترافية.

```txt

```

عرض التقنيات على شكل منظومة تشغيل متكاملة تحتوي على طبقات تقنية مترابطة.

## الهيكل البصري المطلوب

القسم يتكون من 4 أجزاء رئيسية:

1. رأس القسم.
2. خريطة المنظومة التقنية.
3. كروت القيمة.
4. شريط إحصائيات مختصر.

# 1. رأس القسم

في أعلى القسم يتم عرض:

- Badge صغير باسم Technology Ecosystem.
- عنوان عربي كبير وواضح.
- وصف قصير يشرح الفكرة.

التصميم المطلوب:

- محاذاة وسطية.
- خلفية نظيفة وفاتحة.
- تدرج خفيف جدًا بالألوان.
- استخدام اللون الأساسي للموقع مثل teal أو اللون المعتمد حاليًا.
- Typography احترافي.

# 2. خريطة المنظومة التقنية

بدل عرض الشعارات في Marquee فقط، يتم إنشاء شكل يشبه خريطة تقنية / Orbit Map.

في المركز يوجد كرت زجاجي مضيء يحتوي على:

```txt
Smart Agency Stack
```

وحوله بطاقات تصنيفات متصلة به بخطوط ناعمة.

التصنيفات المقترحة:

- Frontend
- Backend
- Mobile
- Database
- DevOps
- AI & Automation

كل تصنيف يحتوي على:

- أيقونة بسيطة.
- اسم التصنيف.
- 2 إلى 4 تقنيات من نفس التصنيف.

مثال:

```txt
Frontend
React - Next.js - TypeScript
```

```txt
Backend
NestJS - Node.js - GraphQL
```

```txt
Database
PostgreSQL - MongoDB - Redis
```

# 4. شريط الإحصائيات

في أسفل القسم يتم عرض شريط إحصائيات بسيط.

المقترح:

```txt
+25 تقنية
+6 طبقات
UI + Backend + DevOps + AI
```

يتم احتساب عدد التقنيات من البيانات القادمة من API:

```ts
const totalTechnologies = technologies.length;
```

ويتم احتساب عدد التصنيفات من البيانات:

```ts
const totalCategories = new Set(technologies.map(item => item.category)).size;
```

إذا كان العدد قليلًا في مرحلة التطوير، يمكن إظهار النص بطريقة مرنة:

```txt
تقنيات متعددة
طبقات متكاملة
واجهة + أنظمة + تشغيل + ذكاء
```

# السلوك التفاعلي المطلوب

## الحالة الافتراضية

عند دخول المستخدم للقسم:

- تظهر المنظومة كاملة.
- يظهر الكرت المركزي.
- تظهر كل التصنيفات حوله.
- يظهر في كل تصنيف أهم 3 أو 4 تقنيات.

## عند Hover على تصنيف

يحدث التالي:

- يضيء كرت التصنيف.
- تصبح خطوط الاتصال الخاصة به أوضح.
- تظهر التقنيات داخله بشكل أوضح.
- يمكن إظهار وصف صغير إن كان التصميم يسمح.

## عند الضغط على تصنيف

اختياري وليس إلزاميًا، لكن يفضل:

- يصبح التصنيف Active.
- يتم عرض قائمة أكبر لتقنيات هذا التصنيف أسفل أو بجانب الخريطة.
- لا يتم تغيير الصفحة.
- لا يتم استدعاء API جديد.

# التعامل مع البيانات الحالية

يجب استخدام نفس البيانات القادمة من API الحالي.

لا تفترض وجود حقول جديدة.

الحقول المتوقعة:

```ts
{
  name: string;
  icon?: string;
  category?: string;
  description?: string;
  tooltip?: string;
}
```

في حال كان `icon` عبارة عن URL، يتم عرضه كصورة.

في حال فشل تحميل الصورة، يتم عرض fallback بسيط مثل أول حرف من اسم التقنية.

مثال:

```tsx
<img
  src={tech.icon}
  alt={tech.name}
  onError={(e) => {
    e.currentTarget.style.display = 'none';
  }}
/>
```

أو استخدام fallback component.

# تحسين مهم في المنطق الحالي

إذا كان الكود الحالي يستخدم ترتيبًا عشوائيًا مثل:

```ts
const shuffled = [...allTechnologies].sort(() => Math.random() - 0.5);
```

يفضل إزالته.

السبب:

- يجعل ترتيب التقنيات يتغير بشكل غير متوقع.
- يسبب تجربة غير مستقرة.
- قد يؤدي إلى اختلافات مزعجة في Render.

الأفضل:

```ts
const sortedTechnologies = [...technologies].sort((a, b) => {
  const categoryCompare = String(a.category || '').localeCompare(String(b.category || ''));
  if (categoryCompare !== 0) return categoryCompare;
  return String(a.name || '').localeCompare(String(b.name || ''));
});
```

# التصميم المطلوب بالتفصيل

## الخلفية

- خلفية فاتحة.
- تدرج خفيف جدًا من الأبيض إلى teal فاتح.
- دوائر Blur خفيفة في الزوايا.
- خطوط أو دوائر Orbit خفيفة خلف الكرت المركزي.

## الألوان

استخدم ألوان المشروع الحالية قدر الإمكان.

المقترح:

```txt
Primary: #008080 أو اللون المعتمد في المشروع
Dark Text: #0F172A
Muted Text: #64748B
Card Background: rgba(255,255,255,0.72)
Border: rgba(0,128,128,0.14)
Glow: rgba(0,128,128,0.18)
```

## الكروت

- Border radius كبير.
- Shadow ناعم.
- خلفية زجاجية.
- Border خفيف.
- Hover لطيف.

## الحركة

يمكن استخدام Framer Motion إن كان موجودًا في المشروع.

الحركات المطلوبة:

- ظهور تدريجي للقسم.
- Floating خفيف للكرت المركزي.
- Hover scale بسيط للكروت.
- خطوط الاتصال لا تحتاج Animation معقد.

تجنب الحركات المبالغ فيها.

# Responsive Design

## Desktop

- الخريطة تظهر كـ Hub-and-spoke.
- التصنيفات موزعة حول الكرت المركزي.
- كروت القيمة تظهر في صف واحد أو Grid من 4 أعمدة.

## Tablet

- الخريطة يمكن أن تصبح Grid من عمودين حول الكرت المركزي.
- كروت القيمة 2 × 2.

## Mobile

على الموبايل لا يفضل استخدام خريطة معقدة جدًا.

المطلوب:

- الكرت المركزي في الأعلى.
- التصنيفات تظهر ككروت رأسية أو Grid من عمود واحد.
- يتم إخفاء خطوط الاتصال المعقدة.
- كروت القيمة تظهر Stack عمودي.
- شريط الإحصائيات يصبح عموديًا أو Scroll أفقي خفيف.

مثال CSS منطقي:

```css
@media (max-width: 768px) {
  .tech-connection-lines {
    display: none;
  }
}
```

```tsx
function Technologies() {}
```

# قواعد مهمة للتنفيذ

1. لا تعدل API.
2. لا تضف حقولًا إلزامية جديدة للباك إند.
3. لا تكسر عرض البيانات الحالية.
4. لا تجعل التصميم يعتمد على أسماء تقنيات ثابتة فقط.
5. استخدم البيانات القادمة من API.
6. يمكن إضافة Mapping داخلي للتصنيفات في Front-end.
7. اجعل التصميم متوافقًا مع RTL.
8. اجعل النصوص العربية واضحة ومصححة.
9. احذف أو قلل الاعتماد على Marquee التقليدي.
10. حافظ على سرعة تحميل القسم.

# المطلوب النهائي من كودكس

تنفيذ إعادة تصميم قسم التقنيات في الصفحة الرئيسية ليصبح:

- قسمًا عصريًا باسم Technology Ecosystem.
- يحتوي على خريطة منظومة تقنية حول Smart Agency Stack.
- يعرض التصنيفات التقنية كطبقات مترابطة.
- يعرض كروت قيمة توضح فائدة التقنيات.
- يعرض شريط إحصائيات مختصر.
- يستخدم البيانات الحالية من API.
- لا يعدل الباك إند نهائيًا.
- يكون Responsive بشكل ممتاز.
- يحافظ على هوية الموقع وألوانه.

# معايير القبول

يعتبر التنفيذ ناجحًا إذا تحقق التالي:

- القسم لم يعد مجرد Marquee أو Grid تقليدي.
- التصميم يظهر كقسم احترافي مناسب لوكالة برمجية.
- البيانات الحالية تظهر بدون أخطاء.
- التصنيفات تظهر بوضوح.
- القسم يعمل على Desktop و Mobile.
- لا توجد تعديلات Backend.
- لا توجد أخطاء TypeScript.
- لا توجد أخطاء Console بسبب الصور أو البيانات الناقصة.
- التصميم متوافق مع الاتجاه العربي RTL.
